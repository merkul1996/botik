# Анализ проекта НейроСпутник

Полный разбор: ошибки, недоделки, привлечение пользователей, перспективы.

**Исправления внесены:** безопасность (Telegram initData), Postgres (профиль, реферал, стрики, триал, active_date, last_chat), учёт бонусных сообщений в лимите, обработка JSON от AI, фронт (initData, кастомные персонажи), тесты storage.

---

## 1. Критические ошибки и баги

### 1.1 Безопасность: userId не проверяется

- **Файл:** `src/server.js`
- **Суть:** Функция `validateTelegramInitData(initData)` реализована, но **нигде не вызывается**. Все API принимают `userId` из тела/query от клиента.
- **Риск:** Любой может отправить чужой `userId` и читать/писать чаты, тратить лимиты, применять реферальные коды от имени другого пользователя, менять план и т.д.
- **Что делать:** В middleware или в каждом обработчике проверять подпись Telegram (`initData`), извлекать `user.id` из неё и использовать только его. Не доверять `req.body.userId` / `req.query.userId` без проверки.

### 1.2 Postgres: профиль без реферального кода и бонусов

- **Файл:** `src/storage.js`
- **Суть:**
  - `ensureUserProfile()` при создании профиля делает только `INSERT (user_id, plan)`. Поле `referral_code` не заполняется — у новых пользователей в Postgres он всегда `NULL`.
  - В `ensureUserProfile()` в SELECT указаны только `user_id, plan`. Поля `referral_code`, `bonus_messages`, `trial_ends_at` не выбираются, поэтому в ответе API они будут `undefined`/0 даже если в БД есть значения.
- **Следствие:** В режиме Postgres реферальная механика и отображение бонусных сообщений не работают; код «Пригласи друга» бесполезен.
- **Что делать:**
  - При первом создании профиля генерировать и записывать `referral_code` (аналог `generateReferralCode()` в memory).
  - В SELECT профиля добавить `referral_code`, `bonus_messages`, `trial_ends_at` (и при необходимости `referred_by`).

### 1.3 Postgres: бонусы за стрик не начисляются

- **Файл:** `src/storage.js`, функция `updateStreak()`
- **Суть:** В режиме memory при стрике 7 и 30 дней начисляются бонусные сообщения (`profile.bonusMessages += 10/30`). В ветке для Postgres только обновляется таблица стриков, **начисление бонусов в `user_profiles.bonus_messages` отсутствует**.
- **Следствие:** В Postgres пользователи не получают обещанные бонусы за 7 и 30 дней.
- **Что делать:** После обновления стрика в Postgres при `currentStreak === 7` и `currentStreak === 30` делать `UPDATE user_profiles SET bonus_messages = bonus_messages + 10/30 WHERE user_id = $1`.

### 1.4 Триал Pro: нигде не выставляется

- **Суть:** В онбординге написано «3 дня Pro бесплатно», в памяти по умолчанию у нового пользователя `plan: "pro"` и `trialEndsAt`. В коде нет логики выдачи триала новому пользователю в Postgres и нет единого потока «первый вход → выдать trial».
- **Следствие:** Обещание «3 дня Pro бесплатно» не выполняется для реальных пользователей (Postgres).
- **Что делать:** Явно ввести «триал при первом визите»: при первом `getUserProfile`/`ensureUserProfile` выставлять `plan = 'pro'`, `trial_ends_at = NOW() + 3 days`, и в `getUserProfile` использовать `resolveTrialPlan` и для Postgres (снижать план после истечения триала и обновлять БД).

### 1.5 Состояние свидания и last_chat только в памяти

- **Файл:** `src/storage.js`
- **Суть:** `getActiveDate`, `setActiveDate`, `clearActiveDate` и `setLastChatPersona`, `getLastChatPersona` работают только с `memoryDates` / `memoryLastChatPersona`. В Postgres нет таблиц и запросов.
- **Следствие:** При перезапуске сервера в Postgres активные свидания «теряются», ревность (кто последняя в чате) не персистится.
- **Что делать:** Либо завести таблицы/поля в Postgres для активного свидания и последнего чата, либо явно описать в доке, что это «сессионное» поведение.

## 2. Недоделки и несоответствия

### 2.1 Лимиты: бонусные сообщения не учитываются

- **Суть:** В профиле есть `bonusMessages`, но в `server.js` лимит проверяется как `usageToday >= limitToday`. Нет логики «сначала списывать бонусные, потом лимит тарифа».
- **Следствие:** Бонусные сообщения из рефералов и стриков по факту не увеличивают доступный лимит.
- **Что делать:** При проверке лимита считать доступные сообщения как `limitToday + bonusMessages - usageToday` (и при отправке уменьшать `bonusMessages` при необходимости).

### 2.2 Подарки: Stars не списываются

- **Файл:** `src/server.js`, `POST /api/gift`; `storage.js` — подарки имеют поле `stars`.
- **Суть:** Подарок записывается, affection растёт, но **оплата Telegram Stars за подарок не инициируется**. Пользователь не платит — получает эффект бесплатно.
- **Что делать:** Либо интегрировать создание инвойса/оплату Stars за подарок и вызывать её перед `recordGift`, либо убрать отображение цены в Stars и сделать подарки бесплатными по дизайну.

### 2.3 Rate limit по userId из тела запроса

- **Файл:** `src/server.js`, `rateLimitMap`
- **Суть:** Ключ лимита: `req.body?.userId || req.query?.userId || req.ip`. При поддельном `userId` можно распределять запросы по разным «пользователям» и обходить лимит.
- **Что делать:** После внедрения проверки Telegram Init Data использовать для rate limit только проверенный `userId` (и при отсутствии — `req.ip`).

### 2.4 SQL: интерполяция в getInactiveUsers

- **Файл:** `src/storage.js`
- **Суть:** `INTERVAL '${hoursThreshold} hours'` — подстановка числа в запрос. Сейчас `hoursThreshold` приходит только из кода (24), но практика опасна.
- **Что делать:** Использовать параметризованный запрос, например `$1::integer` для количества часов и `INTERVAL '1 hour' * $1`.

### 2.5 Фронт: кастомные персонажи не отображаются

- **Суть:** API есть (`/api/custom-personas`, генерация), но в `public/app.js` список персонажей строится только из `personas` с сервера (`/api/personas`). Кастомные персонажи не подмешиваются в сетку и в выбор для чата/свиданий.
- **Что делать:** При загрузке профиля или отдельным запросом получать кастомных персонажей и отображать их в «Персонажах» и в выборе для свиданий (с пометкой «свой»).

### 2.6 Сцены (scenes) в UI не используются

- **Суть:** В `ai.js` есть `scenes` (кафе, ночная прогулка и т.д.), API `/api/scenes` есть. Во фронте используются только `dateScenarios` (парк, кино, пляж и т.д.). Сцены из `scenes` нигде не показываются и не запускаются.
- **Что делать:** Либо завести отдельный режим «ролевая сцена» по выбранной сцене, либо убрать сцены/API, чтобы не вводить в заблуждение.

### 2.7 Ошибки парсинга JSON от OpenAI

- **Файл:** `src/ai.js` — `generatePersona`, `generateDateScene`
- **Суть:** `JSON.parse(raw)` без try/catch. При невалидном ответе модели сервер падает с необработанным исключением.
- **Что делать:** Обернуть в try/catch, логировать `raw`, возвращать пользователю понятное сообщение и при возможности повторять запрос или использовать fallback.

### 2.8 Нет тестов и CI

- **Суть:** В `package.json`: `"test": "echo \"No tests yet\""`. Нет автотестов, нет CI.
- **Что делать:** Минимум — юнит-тесты для storage (memory + при желании postgres), для ключевых API (чат, профиль, реферал). Затем добавить GitHub Actions (или аналог) для прогона тестов.

## 3. Что добавить для привлечения пользователей

### 3.1 Продукт и контент

- **Вирусный онбординг:** После первого сообщения — «Поделись с другом» с готовой кнопкой «Отправить в Telegram» (pre-filled текст + ссылка на бота). Измерить конверсию в переход и в регистрацию.
- **Один бесплатный подарок в день** (или первый подарок бесплатно): снижает порог входа, знакомит с механикой и с реакцией персонажа.
- **Ежедневная «фраза дня» или лёгкий квест** (например, «получи комплимент от Луны») с бонусом сообщений — даёт повод заходить каждый день и повышает стрики.
- **Глубже про «память»:** Явно показывать в UI «Она знает о тебе» (список фактов) и подсказку «Напиши "запомни: …"». Это сильное отличие от обычных чат-ботов.
- **Короткие превью персонажей:** На карточке или в отдельном экране — 1–2 цитаты-примера ответов в стиле персонажа, чтобы пользователь сразу понимал тон.

### 3.2 Монетизация и удержание

- **Чётко показать лимит:** «Осталось X из Y сообщений сегодня» всегда на виду (в шапке чата или под полем ввода). Предупреждение за 5 сообщений до лимита.
- **Триал Pro реально выдать:** 3 дня Pro при первом входе (см. п. 1.4). После окончания — один экран «Твой триал закончился», сравнение Free vs Pro и кнопка «Продолжить с Pro».
- **Реферальная мотивация:** Не только +5 сообщений, но и «приведи 3 друзей — получи неделю Pro» (или скидку в Stars). Показывать счётчик приглашённых в профиле.

### 3.3 Каналы и контент

- **Канал/группа бота:** Короткие посты про персонажей, обновления, «фраза дня», мемы. Ссылка в описании бота и в онбординге.
- **Telegram Ads / таргет:** Таргет на русскоязычную аудиторию по интересам (отношения, развлечения, игры). A/B тесты креативов с разными персонажами.
- **Партнёрства:** Обзоры в каналах про ботов/мини-приложения, разовые коллабы с инфлюенсерами (один пост + ссылка).
- **ASO и описание бота:** В BotFather — понятное описание, ключевые слова (AI-подруга, персонажи, свидания, память). Иконка и превью Mini App — по одному стилю с карточками персонажей.

### 3.4 Метрики

- **Минимум:** Регистрации (первый /start или первый запрос в Mini App), DAU/MAU, конверсия в Pro (или в первую оплату), удержание D1/D7. Хранить события в БД или отправить в аналитику (например, Amplitude/PostHog или свой лог).
- **Рефералы:** Сколько пользователей пришли по реферальному коду и сколько из них остались. Так можно оценить качество канала «друзья».

## 4. Перспективы развития

### 4.1 Краткосрочно (1–3 месяца)

- **Исправить критические баги** из раздела 1 (безопасность, Postgres-профиль, стрики, триал, при необходимости персистентность свиданий и last chat).
- **Учёт бонусных сообщений** в лимите и списание бонусов (раздел 2.1).
- **Реализовать обещанный триал** и показ лимита (раздел 3.2).
- **Добавить кастомных персонажей во фронт** (раздел 2.5).
- **Реферальная программа** с понятным описанием и счётчиком приглашённых (и исправлением реферального кода в Postgres).

После этого продукт предсказуемо работает, монетизация и виральность не ломаются из-за багов.

### 4.2 Среднесрочно (3–6 месяцев)

- **Голосовые сообщения:** Запись голоса пользователя → текст (STT) → ответ модели → TTS от имени персонажа. Сильное отличие в Telegram-нише.
- **Фото в чат:** Загрузка фото (например, «что я сегодня надела») → описание через vision-модель → ответ в характере персонажа.
- **Расширенная память:** Не только «запомни: …», но и авто-вывод фактов из диалога (имя, город, работа, предпочтения) с согласием пользователя («Запомнить, что тебя зовут …?»).
- **Ещё персонажи или «сезоны»:** Новые героини или временные события (праздники, темы), чтобы возвращать старых пользователей и привлекать новых.
- **Подписка с автопродлением:** Не только разовая оплата 200 Stars, но и подписка (если платформа поддерживает), с напоминаниями об окончании.

### 4.3 Долгосрочно (6–12 месяцев)

- **Мультиязычность:** Английский (и при успехе — ещё языки) для выхода на глобальный рынок Telegram.
- **Групповые/парные сценарии:** «Свидание с двумя подругами» или сценарии с выбором «кого ревнует Луна» — больше игровых механик.
- **UGC:** Пользователи создают и публикуют сценарии свиданий или шаблоны персонажей (с модерацией), рейтинг и каталог.
- **Мобильное приложение (PWA или нативное):** Для тех, кто хочет использовать сервис вне Telegram; единый бэкенд уже есть.

## 5. Итоговая сводка

| Категория | Что сделано | Что критично доделать |
|-----------|-------------|------------------------|
| **Безопасность** | Rate limit, CORS | Проверка Telegram Init Data, не доверять client userId |
| **Хранилище (Postgres)** | Таблицы, чат, факты, стрики, рефералы | referral_code при создании и в SELECT; бонусы за стрик; триал |
| **Лимиты и бонусы** | Дневные лимиты по тарифу | Учёт bonusMessages в лимите и списание бонусов |
| **Подарки** | Реакция и XP | Оплата Stars или явно «подарки бесплатные» |
| **Фронт** | Чат, свидания, профиль, онбординг | Кастомные персонажи в списке; сцены либо использовать, либо убрать |
| **Привлечение** | Реферальная механика (сломана в Postgres) | Триал, вирусный онбординг, лимит на виду, метрики |
| **Качество кода** | Рабочий MVP | Тесты, CI, обработка ошибок AI, параметризованный SQL |

Главный вывод: продукт по идее и фичам сильный (персонажи, память, свидания, ревность, рефералы), но **без проверки userId и исправления Postgres-логики** он небезопасен и часть обещаний пользователям (рефералы, стрики, триал) не выполняется. Приоритет — безопасность и корректная работа профиля и бонусов в Postgres, затем — привлечение и перспективные фичи.
